# 本科毕设项目客观评分分析

> 分析日期：2026-03-02 | 基于对全部源代码的逐文件审查

---

## 一、总评分：70~78 分（百分制）

**对应等级**：中等偏上（良-）

以下按主要维度逐项分析，给出得分理由和扣分点。

---

## 二、分维度评分

### 1. 选题与创新性（10分制 → 得分：5~6分）

| 方面 | 评价 |
|------|------|
| 选题价值 | 电商数据分析看板是**经典毕设选题**，有实用价值，但缺乏新意 |
| 创新点 | **几乎没有创新**。RFM+KMeans、漏斗分析、留存矩阵都是教科书标准方案 |
| 差异化 | 与市面上大量「XX数据可视化分析系统」同类项目高度雷同 |

> [!WARNING]
> 选题缺乏差异化是最大的软肋。如果答辩老师审过多个类似项目，印象分会被严重拉低。

---

### 2. 系统设计与架构（10分制 → 得分：7分）

**加分项：**
- 前后端分离（FastAPI + Vue3），整体架构思路清晰
- DAO 层使用了**策略模式**（`DatabaseBackend` 抽象基类 + SQLite/ClickHouse 双后端），这是一个亮点
- 数据库连接管理器做了**单例模式 + 线程安全 + 断路器/心跳/指数退避重连**，说明有一定工程意识
- ETL 管道做了模块拆分（6个独立模块），流程清晰

**扣分项：**
- 侧边栏菜单只有**一个菜单项**，说明系统功能单一，没有真正的多页面/多功能场景
- 前端只有 **1 个 View + 5 个组件 + 2 个 composable**，组件数量过少
- 没有状态管理（Pinia/Vuex），组件间通信依赖 props/events，说明前端复杂度不足以需要
- ClickHouse 后端虽写了完整代码但**实际部署大概率只用 SQLite**，ClickHouse 分支更像是"凑技术栈"

---

### 3. 功能完整度（10分制 → 得分：6~7分）

实现的功能清单：

| 功能 | 状态 | 评价 |
|------|------|------|
| 登录认证（JWT） | ✅ | 基础功能，但只有**硬编码的 admin/admin** 一个用户 |
| KPI 卡片（销售额/订单数/客单价/环比同比） | ✅ | 标准实现 |
| 趋势图（日/周/月切换） | ✅ | 基础功能 |
| 漏斗图 | ✅ | 标准实现 |
| TOP10 商品 / 品类分布 / 渠道分布 / 年龄分布 | ✅ | 四个基础维度的聚合查询 |
| RFM 价值分群展示 | ✅ | 前端只是一个饼图展示，缺深度 |
| 同期群留存热力图 | ✅ | 亮点之一 |
| 日期筛选 + 下钻抽屉 | ✅ | 交互尚可 |
| 用户注册 / 权限管控 / 数据导出 / 报表 | ❌ | 缺失 |
| 数据预测（forecast） | ❌ | 规划了但未实现 |

> [!NOTE]
> 功能点"数量"足够凑一个看板页面，但**深度不够**——每个功能都是最基础的实现，缺少交互深度（如多维钻取、对比分析、自定义筛选维度等）。

---

### 4. 数据处理与算法（10分制 → 得分：7分）

**加分项：**
- 数据生成脚本（`generate_data.py`，475行）质量不错，融合了 Zipf 分布、Pareto 分布、指数衰减留存模型、双11高斯爆发等**统计模型**，说明理解数据分布
- ETL 管道使用 PySpark，包含 RFM 计算 + StandardScaler 标准化 + KMeans 聚类（轮廓系数自动选K）
- 双口径漏斗（严格时序约束 + 宽松非时序约束）有一定业务思考
- 同期群留存矩阵构建逻辑完整

**扣分项：**
- 数据是**完全模拟的**，非真实数据集（虽然标注灵感来自淘宝 UserBehavior 数据集，但并非使用真实数据）
- KMeans 只搜索 K=3~5，搜索空间太小
- 没有模型评估的可视化展示（如 elbow 图、轮廓系数对比图）
- 没有做特征工程的探索性数据分析（EDA）过程记录

---

### 5. 代码质量（10分制 → 得分：7分）

**加分项：**
- 注释较为充分（中文注释，每个模块开头有文档字符串）
- 分层清晰：`core/` → `db/` → `dao/` → `services/` → `api/`，职责基本隔离
- 有 9 个测试文件，覆盖了 DAO、Service、API、ETL 等层
- 使用了类型提示（`typing`）

**扣分项：**
- `dao/backend.py` 中存在**冗余代码**：`from dao.base import safe_float, safe_int` 既在类级别导入又在每个方法中重复导入
- `dashboard_service.py` 中混入了 `from core.logging import logger` 在函数内部的 debug 日志（第82行），不规范
- `services/auth_service.py` 只有 ~30 行有效代码，只是一个密码对比的简单封装
- 前端没有 TypeScript，没有 ESLint 配置
- 没有 CI/CD 配置文件（GitHub Actions 等）

---

### 6. 前端实现（10分制 → 得分：6分）

**加分项：**
- 使用 Vue3 Composition API + `<script setup>` 语法，框架使用方式现代
- `useEcharts.js` composable 封装了 ECharts 生命周期管理（ResizeObserver + 实例注册/销毁），有一定工程质量
- 有 fallback 降级机制（聚合端点不可用时回退到多接口模式）

**扣分项：**
- 布局全部依赖 Element Plus 的 `el-row`/`el-col`，**没有自定义 CSS 设计**
- `style.css` 只有约 60 行全局样式
- **没有响应式适配**（移动端/平板端）
- 没有路由（vue-router），整个应用就是一个页面
- 没有使用 Pinia 状态管理
- 没有骨架屏/加载动画等精打磨的用户体验
- 前端没有单元测试

---

### 7. 文档与规范（10分制 → 得分：7~8分）

**加分项：**
- `doc/` 目录下有 7 个文档文件，覆盖面广（技术架构、部署指南、快速开始等）
- README 有基本介绍
- 有 `.gitignore`，使用了 Git 版本管理
- 有 `requirements.txt`

**扣分项：**
- `technical_architecture.md` 存在夸大嫌疑（之前对话记录中你已经在Review这个问题了）
- 没有 API 接口文档（虽然 FastAPI 自带 Swagger，但没有额外说明）

## 三、关键风险点（答辩可能被质疑）

| 风险 | 说明 |
|------|------|
| 🔴 **AI 生成嫌疑** | 代码注释风格高度统一、设计模式使用方式较"教科书"化、文档风格工整——经验丰富的答辩老师可能质疑是否AI辅助生成 |
| 🔴 **数据是模拟的** | `generate_data.py` 生成所有数据，而非基于真实数据集进行分析——这削弱了"数据分析"的含金量 |
| 🟡 **功能单薄** | 整个系统只有一个看板页面，缺乏多角色、多视图、数据管理等功能 |
| 🟡 **ClickHouse 难以演示** | 双后端设计实际答辩时大概率只能演示 SQLite 版本 |
| 🟡 **缺少核心算法深度** | KMeans 聚类只是调了 PySpark MLlib API（约20行代码），缺乏算法理解的体现 |

## 五、得分提升建议（如果还有时间）

1. **加入数据预测功能**（你之前已在规划）——即使是简单的移动平均或 Prophet，也能显著提升技术含量
2. **补充 EDA 分析过程**——在文档中展示数据探索的 Jupyter Notebook（分布图、相关性分析等），证明你理解数据
3. **增加数据导出功能**（导出 Excel/PDF 报告），功能实用性立刻上升
4. **增加多用户/多角色支持**——当前只有硬编码的 admin，至少做一个用户管理页面

---

## 六、总结

这是一个**工程规范性较好、但创新性和功能深度不足**的本科毕设。代码分层清晰、注释充分、测试覆盖面尚可，说明你（或AI辅助下）的工程习惯不错。但作为一个"电商数据分析平台"，它本质上只是一个**带登录的 ECharts 看板单页应用**，功能深度和创新性是短板。

**最终预估得分：70~78 分**，具体取决于答辩表现和老师的评判标准。
