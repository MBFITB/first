## 选项一：标准版（适合空间充足，全面展示技术深度，推荐使用！）

项目名称：电商数据全景分析系统 （全栈独立开发）  
核心技术：PySpark / ClickHouse / FastAPI / Vue 3 / ECharts / 设计模式  
项目描述：独立开发覆盖“底层数据模拟 → ETL 特征计算 → 高可用后端服务 → 响应式数据展现大屏”的全链路商业全景数据洞察系统。  
核心职责与亮点：  
- 大数据 ETL 多维计算：基于 PySpark 搭建 6 大模块化离线分析引擎，深度处理约 200 万条（包含日内潮汐、指数留存等真实运营约束）的电商流水；实现了“重时序严约束”与“重行为宽约束”的双口径转化漏斗及灵活维度的同期群（Cohort）留存矩阵。
- 算法赋能与人群打标：整合数据标准化与 KMeans 聚类算法，配合模型轮廓系数进行 K 值自动寻优（K=3~5），基于 R/F/M 面向海量用户实现价值加权分数打标，精准筛选出“核心高价值/沉睡流失/高频”四大客群。
- 高可用架构与双存储容灾：针对大数据看板并发与可用性诉求挑战，创新设计 ClickHouse（性能优先）与 SQLite（本地降级）双存容灾；通过策略模式与抽象基类自底向上剥离数据库异构差异；在业务底层手写搭载指数级网络退避、10s 心跳判定与 60s 自动断路器的流量熔断保护机制。
- 并发优化与前端交互响应：基于 Vue3 与 ECharts 开发 8 类复合交互看板（含点击钻取分析），自研全局渲染池实现容器自适应缩放；由服务接口端结合连接池优化将看板所需 8 个零碎高频图表请求打包合并为 1 次单一聚合调用，极大减少全景渲染白屏时间与客户端网络请求开销。
- 严谨的企业级工程实践：落地 ClickHouse 新旧业务表无损原子交换，确保查询无脏读；开发全链路状态监听捕捉数据异常丢失率并生成质量报告；全程介入基于中间件的统一 JWT 高精度白名单鉴权，并配套覆盖测试。
- AI 智能数据分析引擎：基于大语言模型（DeepSeek）自主研发“Text-to-SQL + 深度业务解读”双模式助手。通过精细化 System Prompt 工程突破模型推理质量瓶颈；在后端运用正则表达式保障生成 SQL 的安全边界（拦截修改操作、强制 LIMIT）；前端结合 Marked.js 与 SSE 协议实现支持 Markdown 富文本渲染的打字机流式输出，极大提升非技术人员的数据挖掘体验与响应感知速度。

## 选项二：紧凑版（适合空间有限，侧重突出微架构和业务价值）

项目名称：全栈电商数据分析决策系统 （全栈独立开发）  
技术栈：PySpark、ClickHouse、Vue 3、FastAPI  
内容及亮点：
- 全栈链路开发构建：独立设计并完成从脏数据模拟、数据并发清洗至前端聚合大屏的端到端实现。完成百万级用户数据流水的挖掘与透视（产出留存、多端占比、类目排行）。
- 机器学习降维应用：应用 KMeans 算法及轮廓系数在全站范围自发建立依据业务量化标准的动态分层模型，协助商业决策判定高价值用户画像。
- 容灾组件与高可用演进：手写封装含探活与指数级重连衰退特征的断路器。依托内部策略设计模式，在主干数据节点失效崩溃时能零介入无缝秒切本地持久化引擎兜底并报警，避免大屏全局故障。
- 前后端渲染链路优化：通过对图表微服务并发提取请求进行合并降维计算压缩响应时间；接入混合周期管理防内存泄露；全面拦截控制层安全鉴权并保障零延迟原子级写库落盘，体系化引入严谨的数据工程理念。
- 引入 LLM 赋能数据检索：集成大语言模型，落地自然语言转 SQL 与自动数据洞察功能。通过前后端流式交互（SSE）与 Markdown 渲染，打造对非技术人员极度友好的智能对话分析体验。

> 💡 面试官可能会追问的技术重点（请务必熟悉你的思路应对答辩）：
>
> 1. 你这个项目的断路器是怎么实现的？  
>    回答思路：我的连接管理器是一个单例。当重试连接数据库多次彻底失败后，会在内部设置一个 60 秒的暂挂时间标记，并强行返回降级后的稳定备用连接。在这 60 秒内即使有千次并发请求打进来，引擎也会直接走回退逻辑保护链路；只有 60 秒过后（半开状态）才去再次尝试发起重连。
> 2. 你设计的原子级表交换解决了什么痛点？  
>    回答思路：因为离线计算常常是全量覆写或重算，传统的先清空后插入，或者删旧表再建新表，往往在这几十秒间隙会让用户从大屏上查不到数据或是查到断层脏数据。我在数据写入环节采用了先将结果写入一张临时新表，再利用底层无损交换语法安全对调旧表的方式，保障业务看板查询始终不中断。
> 3. 讲一下你代码里如何实现模块解耦（怎么使用策略模式的）？  
>    回答思路：我在底层数据访问层并没有写出硬编码判定分支的臃肿代码，而是设计了公用的抽象基类，然后分别让轻量级降级访问类和高性能分析访问类按照同一套标准实现了所有动作。外部服务层面只管调接口而不关心底层走的是哪条路径，实现了开闭原则，即使未来接入新引擎也无需动业务逻辑。
> 4. 双口径转化漏斗是什么，有什么业务意义？  
>    回答思路：在做数据转换计算漏斗时，我不仅追踪了“严格顺序（先浏览再加车再买且时间依次递增）”链路，也抓取了非顺序性或直接跨越环节（宽松逻辑）的离散购买。两者的对比缺口往往能够反映特殊活动的直接秒购、被推广吸引的补单或纯冲动消费行为，这能比仅仅看单一路线漏斗为运营团队提供更丰富的归因与复盘支持。
> 5. 你的 AI 助手是如何解决大模型响应慢和“幻觉”问题的？
>    回答思路：为了解决“慢”，我早期尝试过 SSE 流式推流配合打字机渲染，用 HTTP 长连接让用户第一秒就能看到字，大幅缩减了心智等待时间；此外因为后端做了合并缓冲（节流机制），解决了前端频繁响应式重绘的卡顿。关于防“幻觉”和越权，我在后端拦截层用正则对生成的 SQL 强行卡死只能由 `SELECT` 打头，并且自动拼装 `LIMIT` 保护内存；在 Prompt 层详细投喂了只限定在我这四张数据表上的说明体系，并引导其基于近期历史查询出来的数据直接做文本回复，避免它要求用户“重新去查询别的”。
